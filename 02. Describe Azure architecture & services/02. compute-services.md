# AZ-900 Study Guide: Module 5 — Azure Compute Services

This is a big module — it covers every major way to run workloads in Azure, from full control (VMs) to zero infrastructure management (Functions). Pay close attention to when you'd choose each option, since the exam loves scenario-based questions here.

## 1. Azure Virtual Machines (VMs)

**What it is (Simple Definition):**
Azure VMs let you run virtualized servers as infrastructure as a service (IaaS). Just like a physical server, you control the operating system and whatever software you install on it.

**The Everyday Analogy:**
It's like renting an empty apartment. The building (physical hardware) is maintained for you, but once you're inside, you decide what furniture to bring, how to arrange it, and how to maintain your own space.

**Why it matters (Key Benefit):**
- Total control over the operating system.
- Freedom to run custom software and hosting configurations.
- No need to buy or maintain physical server hardware — though you still handle patching and updates inside the VM.
- Fast deployment from prebuilt images (templates that already include an OS and tools like web hosting components).

**Common use cases:**
- **Testing and development:** Spin up different OS/app configurations quickly, then delete when done.
- **Cloud application hosting:** Run apps and scale capacity up or down as demand changes.
- **Datacenter extension:** Extend an on-premises network into Azure.
- **Disaster recovery:** Keep failover capacity ready in Azure in case your primary site goes down.
- **Lift and shift migration:** Move existing server workloads with minimal redesign.

### Sizing a VM

When provisioning a VM, you choose:
- **Size** (purpose, processor cores, RAM)
- **Storage disks** (HDDs, SSDs, etc.)
- **Networking** (virtual network, public IP, port configuration)

**VM family cheat sheet:**

| Family | Typical Focus | Example Use |
|---|---|---|
| **B-series** | Burstable, cost-efficient | Dev/test workloads with occasional CPU spikes |
| **D-series** | General purpose | Web servers, small-to-medium app servers |
| **E-series** | Memory optimized | In-memory databases, analytics workloads |
| **F-series** | Compute optimized | CPU-intensive application tiers |
| **M-series** | Large memory footprint | Large enterprise databases |
| **L-series** | Storage optimized | High-throughput storage and data processing |
| **N-series** | GPU enabled | AI training/inference, graphics workloads |

**Decoding a VM name (example: D2s_v5):**
- **D** = family (general purpose)
- **2** = number of vCPUs
- **s** = supports Premium SSD storage
- **v5** = hardware generation

**Exam Takeaway:** Pick the family that matches your workload behavior first, then fine-tune the size as performance needs grow.

### Scale and Resiliency Options

**Virtual Machine Scale Sets:**
Let you create and manage groups of identical, load-balanced VMs. Instead of manually keeping configurations consistent, scale sets centralize configuration and automatically scale out or in based on demand or schedule, while integrating with load balancing.

**Virtual Machine Availability Sets:**
Improve resiliency by reducing the chance that all your VMs are hit by the same maintenance event or hardware failure. They group VMs by:
- **Update domain:** VMs that can be rebooted together during planned maintenance.
- **Fault domain:** VMs that share a potential power or network failure point.

> **Note:** Availability sets don't add extra cost — you only pay for the VM instances. In regions that support Availability Zones, zone-based designs are usually preferred, since they provide broader failure isolation.

## 2. Azure Virtual Desktop

**What it is (Simple Definition):**
A managed service that lets users securely access Windows desktops and apps from almost any device or location — the desktop and apps live in the cloud, not on the local device.

**The Everyday Analogy:**
It's like accessing your office computer remotely from anywhere, except the "computer" itself never physically exists on your desk — it's a session running in Azure that you connect to.

**Why it matters (Key Benefit):**
- Centralizes desktop and app delivery, integrating with Microsoft Entra ID for identity and access control.
- Reduces data exposure on local devices, since apps and data can remain in the cloud-hosted session.
- Supports both single-session and multi-session Windows experiences.

**When to use it:** When a team needs centralized desktop and app access across distributed users, contractors, or hybrid workers — for example, a support team that needs standardized tools, access policies, and security across every shift.

**Exam Takeaway:** Azure Virtual Desktop is easier to operate than building separate VM-based desktop environments for each user group when many people need consistent, centrally managed desktops.

## 3. Azure Containers

**What it is (Simple Definition):**
Containers let you run multiple isolated instances of an application on a single host, without managing a separate operating system for each one — unlike VMs, which each run their own full OS.

**The Everyday Analogy:**
If a VM is like renting a whole apartment for each tenant, a container is like a set of lockers in a shared room — lightweight, quick to set up, and quick to remove, without needing to build a new room each time.

**Why it matters (Key Benefit):**
- Lightweight and designed to be created, scaled, and stopped dynamically.
- Faster to restart after a crash or interruption compared to VMs.
- One of the most popular container engines is Docker, which Azure supports.

**Azure's container options:**

| Service | What It Is | Best For |
|---|---|---|
| **Azure Container Instances** | The fastest, simplest way to run a container without managing VMs (PaaS) | Quick, simple container deployment |
| **Azure Container Apps** | Similar to Container Instances, with built-in load balancing and scaling (PaaS) | Apps that need to adapt to changing demand |
| **Azure Kubernetes Service (AKS)** | A container orchestration service that manages the lifecycle of many containers | Managing a large fleet of containers |

**Why use containers:**
Containers are commonly used for **microservice architecture** — breaking an application into smaller, independent pieces (for example, separate containers for front end, back end, and storage). This means you can scale, update, or maintain each piece independently.

> **Practical example:** If your website's back end reaches capacity while the front end and storage are fine, you can scale just the back end — without touching the rest of the application.

**Exam Takeaway:** Containers virtualize the application, not the operating system — that's the key difference from VMs.

## 4. Azure Functions

**What it is (Simple Definition):**
Azure Functions is an event-driven, serverless compute option. Instead of keeping a VM or container running constantly, an event "wakes up" your function to run, and it deallocates automatically when finished.

**The Everyday Analogy:**
Think of a motion-sensor light. It doesn't stay on all the time — it switches on only when triggered by movement, then turns off again. You're only "paying" for the moment it's actually on.

**Why it matters (Key Benefit):**
- You're only billed for the actual CPU time your function uses while running.
- Ideal for quick tasks (seconds or less) triggered by an event, timer, or message from another Azure service.
- Scales automatically based on demand — great for variable workloads.

**Stateless vs. stateful:**
- **Stateless (default):** Behaves as if it restarts fresh every time it responds to an event.
- **Stateful (Durable Functions):** The runtime passes context through the function to track prior activity.

**Exam Takeaway:** Azure Functions removes the need to keep infrastructure "always on" — you focus purely on the code, and Azure handles provisioning and deprovisioning around each event.

## 5. AI, Machine Learning, and IoT/Edge Services

**What it is (Simple Definition):**
Azure provides prebuilt and customizable services so you can add intelligent features or connect physical devices without building everything from scratch.

**The Everyday Analogy:**
Think of Azure AI services like buying a ready-made recipe kit — you get pre-measured ingredients (prebuilt AI capabilities) and just follow the steps. Azure Machine Learning is more like building your own recipe from raw ingredients when you need something fully custom.

### Azure AI Services

- Provides prebuilt capabilities for common AI scenarios: language, speech, vision, and document processing.
- Useful when you want intelligent features through APIs, without training your own model.
- **Azure OpenAI Service** supports generative AI scenarios like chat and content generation, with built-in security and governance.

**Agentic AI patterns:** Applications that combine an AI model with instructions, context, and tool use to complete multistep goals. These are typically built by combining Azure AI services and Azure OpenAI Service with your own application logic — think of it as an application pattern, not a separate compute category.

### Azure Machine Learning

Use this when your scenario requires building, training, and managing **custom** machine learning models — including model development, experimentation, and lifecycle management.

### IoT and Edge Services

| Service | What It Does |
|---|---|
| **Azure IoT Hub** | Enables secure, bi-directional communication between cloud services and IoT devices |
| **Azure IoT Central** | A simplified SaaS IoT platform for solution builders |
| **Azure IoT Edge** | Extends cloud capabilities to edge devices so workloads can run closer to where data is generated |

> **Typical flow:** Devices send telemetry through IoT Hub, cloud analytics generate insights and model updates, and IoT Edge applies those updates near the devices. IoT Central can simplify managing the whole connected-device scenario.

**Exam Takeaway (decision pattern):**
- Need prebuilt AI features via API? Use **Azure AI services**.
- Need custom model development? Use **Azure Machine Learning**.
- Solution centers on connected devices and telemetry? Use **Azure IoT services**.

## 6. Application Hosting Options

**What it is (Simple Definition):**
Beyond VMs and containers, Azure offers additional ways to host applications without managing the underlying infrastructure yourself — the most notable being **Azure App Service**.

**The Everyday Analogy:**
If VMs are like renting an empty apartment and containers are like shared lockers, App Service is like moving into a fully serviced apartment — you bring your belongings (your code), and the building handles maintenance, security, and utilities for you.

### Azure App Service

**What it is:** A platform for building and hosting web apps, background jobs, mobile back-ends, and RESTful APIs in the programming language of your choice, without managing infrastructure.

**Why it matters (Key Benefit):**
- Automatic scaling and high availability.
- Supports both Windows and Linux.
- Supports automated deployments from GitHub, Azure DevOps, or any Git repo.
- Supports multiple languages and frameworks: .NET, .NET Core, Java, PHP, Python, and Node.js.

**Types of App Service apps:**

| Type | What It's For |
|---|---|
| **Web apps** | Hosting websites using ASP.NET, ASP.NET Core, Java, Ruby, Node.js, PHP, or Python, on Windows or Linux |
| **API apps** | Building REST-based web APIs, with full Swagger support and Azure Marketplace publishing |
| **WebJobs** | Running background programs or scripts (.exe, Java, PHP, Python, Node.js, .cmd, .bat, PowerShell, Bash) on a schedule or trigger |
| **Mobile apps** | Building a back end for iOS and Android apps — cloud-based SQL storage, social provider authentication, push notifications, and custom back-end logic in C# or Node.js |

**Shared benefits across all App Service types:**
- Integrated deployment and management.
- Securable endpoints.
- Quick scaling to handle high traffic.
- Built-in load balancing and traffic management for high availability.

**Exam Takeaway:** App Service is the go-to PaaS option for web-oriented applications when you want to focus on your app instead of managing the infrastructure underneath it.

## Quick Module Recap

- **VMs** give maximum control (IaaS); scale sets manage identical VM groups, availability sets protect against localized failures.
- **Azure Virtual Desktop** centralizes desktop/app delivery for distributed users.
- **Containers** virtualize the app, not the OS — lightweight and fast, with Container Instances, Container Apps, and AKS as the main options.
- **Azure Functions** is event-driven and serverless — you pay only for the compute time your code actually uses.
- **Azure AI services** = prebuilt AI via API; **Azure Machine Learning** = custom model building; **IoT services** = connected devices and telemetry.
- **Azure App Service** is the PaaS option for hosting web apps, APIs, WebJobs, and mobile back-ends without managing infrastructure.
