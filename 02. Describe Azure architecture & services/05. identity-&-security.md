# Module 8: Azure Identity, Security, and Governance

This is a dense module covering identity, authentication, access control, and the broader security frameworks Azure uses to protect your environment. These concepts show up throughout the AZ-900 exam, so take your time with each section.

---

## 1. Azure Directory Services (Microsoft Entra ID)

**What it is (Simple Definition):**
Microsoft Entra ID is Microsoft's cloud-based identity and access management service. It lets you sign in and access both Microsoft cloud apps and apps you build yourself.

**The Everyday Analogy:**
Think of Microsoft Entra ID like a single, universal keycard system for a large office campus. Instead of a different key for every door (application), you carry one card that grants access to everything you're authorized to use.

**Why it matters (Key Benefit):**
- If you've used on-premises Active Directory, Entra ID will feel familiar — you still control the identity accounts, but Microsoft ensures global availability.
- Connecting on-premises AD to Entra ID unlocks free suspicious sign-in detection (unexpected locations, unknown devices) that on-premises AD can't do alone.

**Who uses it:**
- **IT administrators:** Control access to apps and resources based on workload and security needs.
- **App developers:** Add standards-based functionality like SSO or support for existing credentials.
- **Users:** Manage their own identity, including self-service password reset.
- **Online service subscribers:** Microsoft 365, Office 365, Azure, and Dynamics CRM Online users already authenticate through Entra ID.

**What it does:**

| Capability | What It Covers |
|---|---|
| **Authentication** | Verifies identity before granting access — includes self-service password reset, MFA, banned password lists, smart lockout |
| **Single sign-on (SSO)** | Lets one identity access multiple applications |
| **Application management** | Manages cloud and on-premises apps via Application Proxy, SaaS integration, My Apps portal |
| **Device management** | Supports device registration through tools like Microsoft Intune, enabling device-based Conditional Access |

### Microsoft Entra Connect

Bridges on-premises Active Directory and Microsoft Entra ID by synchronizing identities between the two. This gives users a consistent experience — SSO, MFA, self-service password reset — whether they're accessing on-premises or cloud resources.

### Microsoft Entra Domain Services

Provides managed domain services (domain join, group policy, LDAP, Kerberos/NTLM authentication) without requiring you to deploy or maintain domain controllers in the cloud.

**Why it matters:**
- Useful for legacy applications that can't use modern authentication — you can lift and shift them into a managed domain.
- Integrates with your existing Entra tenant, so users sign in with existing credentials.

**How it works:**
- You define a unique namespace (domain name); Azure deploys two Windows Server domain controllers (a "replica set") into your chosen region.
- Azure manages the domain controllers entirely, including backups and encryption at rest.
- Synchronization is **one-way**: from Microsoft Entra ID to Microsoft Entra Domain Services. Resources created directly in the managed domain don't sync back to Entra ID.

**Exam Takeaway:** Microsoft Entra Connect syncs between on-premises AD and Entra ID; Microsoft Entra Domain Services provides managed domain controllers in Azure without you having to run them yourself.

## 2. Azure Authentication Methods

**What it is (Simple Definition):**
Authentication proves the identity of a person, service, or device using credentials. Common methods include passwords, SSO, multifactor authentication (MFA), and passwordless sign-in.

**The Everyday Analogy:**
Think of authentication methods like different types of locks on a door. A password is like a basic key — convenient, but easy to copy. MFA is like requiring both a key and a fingerprint scan. Passwordless is like a smart lock that recognizes your face — no key needed at all, and harder to fake.

### Single Sign-On (SSO)

Lets a user sign in once and access multiple trusted applications.

**Why it matters:**
- Reduces password sprawl, lowering credential-related incidents and lockout/reset overhead.
- Simplifies lifecycle management — access tied to one identity is easier to update or remove.

> **Important:** SSO is only as secure as the initial authenticator, since every subsequent connection relies on that first sign-in.

### Multifactor Authentication (MFA)

Requires an extra factor at sign-in, so a stolen password alone isn't enough.

**The three factor categories:**
- **Something you know** — a password or challenge question.
- **Something you have** — a code sent to your phone.
- **Something you are** — a biometric signal like a fingerprint or face scan.

**Microsoft Entra multifactor authentication** is Microsoft's service for this, letting users add a phone call or app notification as a second factor.

### Passwordless Authentication

Eliminates the password entirely, replacing it with a trusted device plus a biometric signal or PIN.

**The three passwordless options Microsoft Entra ID supports:**

| Method | Best For |
|---|---|
| **Windows Hello for Business** | Information workers with a designated Windows PC; credentials tied directly to that device with PKI integration and built-in SSO |
| **Microsoft Authenticator app** | Turning any iOS or Android phone into a sign-in factor — match a number on screen, confirm with biometric or PIN |
| **FIDO2 security keys** | Unphishable hardware devices (USB, Bluetooth, or NFC) that handle authentication with no username or password |

**Exam Takeaway:** Passwordless authentication offers both high security *and* high convenience — a rare combination compared to passwords (high convenience, low security) or MFA alone.

## 3. Azure External Identities

**What it is (Simple Definition):**
An external identity is a person, device, or service outside your own tenant. Microsoft Entra External ID provides the capabilities to securely interact with these outside users.

**The Everyday Analogy:**
Think of it like giving a visiting contractor a temporary visitor badge instead of issuing them a full employee ID. They can access the specific areas they need, using their own existing identification, while you still control what they're allowed to see.

**Why it matters (Key Benefit):**
- Lets partners, suppliers, vendors, and contractors access approved resources using their existing credentials.
- No need to create and manage separate local accounts for every guest.
- External users can bring work accounts, government digital identities, or social identities (like Google or Facebook).

**External ID capabilities:**

| Capability | What It Does |
|---|---|
| **B2B collaboration** | External users sign in with their preferred identity to access your apps; represented in your directory as guest users |
| **B2B direct connect** | Establishes two-way trust with another Entra tenant, currently supporting Teams shared channels; users aren't represented in your directory but are visible in Teams admin reports |
| **Microsoft Entra External ID for customers** (formerly Azure AD B2C) | Publishes modern SaaS or custom apps to consumers, using Entra External ID for identity management |

**Managing guest access over time:**
Guests can be invited by administrators or authorized users. You can also run access reviews, asking guests or a decision maker to recertify continued access, then remove access for guests who no longer need it.

**Exam Takeaway:** External identities cover cross-tenant and consumer access scenarios, distinct from SSO, which is about one identity accessing multiple apps within your own environment.

## 4. Azure Conditional Access

**What it is (Simple Definition):**
Conditional Access is a Microsoft Entra ID tool that allows or denies access to resources based on identity signals — who the user is, where they are, and what device they're using.

**The Everyday Analogy:**
Think of it like a security guard who lets regular employees walk in without a second glance, but stops and asks for extra ID if someone shows up at an unusual hour or from an unfamiliar entrance.

**Why it matters (Key Benefit):**
- Helps IT admins empower users to be productive anywhere, while still protecting critical assets.
- Provides granular MFA — a known location might not trigger a second factor, but an unusual sign-in will.

**How it works:**
1. **Signals** are collected during sign-in (location, device, application being accessed).
2. A **decision** is made based on those signals (allow, block, or require MFA).
3. **Enforcement** carries out the decision.

**Common use cases:**
- Require MFA based on role, location, or network (e.g., for administrators or off-network sign-ins).
- Require access only through approved client applications.
- Require access only from managed, compliant devices.
- Block access from untrusted or unexpected sources.

**Exam Takeaway:** Conditional Access follows a signal → decision → enforcement flow, and it's what enables context-aware MFA challenges rather than a one-size-fits-all rule.

## 5. Azure Role-Based Access Control (Azure RBAC)

**What it is (Simple Definition):**
Azure RBAC lets you control access to resources by assigning roles to people or groups, following the **principle of least privilege** — only granting the access needed to do the job.

**The Everyday Analogy:**
Think of it like giving employees keycards that only open the specific rooms relevant to their job — a cleaner's card opens every room but not the safe, while an accountant's card opens the finance office but not the server room.

**Why it matters (Key Benefit):**
- Built-in roles cover common access scenarios; you can also create custom roles.
- Assigning someone to a role automatically grants the associated permissions — no need to configure access individually.
- When a new person joins a team's RBAC group, they automatically inherit the same access as their peers.

**Scopes** — the resource or set of resources a role applies to:
- Management group (a collection of subscriptions)
- A single subscription
- A resource group
- A single resource

**Why it matters:**
- Azure RBAC is **hierarchical** — permissions granted at a parent scope are inherited by all child scopes. For example, an Owner role at the management group level applies to everything in every subscription underneath it.
- Azure RBAC is enforced on any action passing through **Azure Resource Manager** (accessed via the portal, Cloud Shell, PowerShell, or CLI) — it does not enforce access at the application or data level; that's the application's responsibility.
- Azure RBAC uses an **allow model** — permissions from different role assignments combine, so read access from one assignment plus write access from another gives you both.

**Exam Takeaway:** Azure RBAC permissions are additive and inherited downward through the hierarchy — resources, resource groups, subscriptions, and management groups.

## 6. Zero Trust Model

**What it is (Simple Definition):**
Zero Trust assumes breach from the outset and verifies every request as though it came from an uncontrolled network — regardless of where that request originates.

**The Everyday Analogy:**
Traditional security is like a castle with a moat — once you're inside the walls, you're trusted. Zero Trust is more like an airport, where everyone is screened at every checkpoint, no matter how many times they've passed through before.

**Guiding principles:**
- **Verify explicitly:** Always authenticate and authorize using all available data points.
- **Use least privilege access:** Limit access with Just-In-Time and Just-Enough-Access, risk-based policies, and data protection.
- **Assume breach:** Limit potential impact, segment access, verify end-to-end encryption, and use analytics for visibility and threat detection.

**Why it matters:**
- Traditional perimeter-based networks assumed anything inside the network was safe.
- Zero Trust instead grants access based on authentication and context, not network location.

> **Practical example:** A user signing in from an unmanaged device on a public network is still evaluated on identity signals, device state, and risk. Access might be allowed for low-risk apps, require MFA for sensitive systems, or be blocked entirely if risk is too high.

**Exam Takeaway:** Zero Trust access decisions are continuous and context-aware — never based solely on where the request comes from.

## 7. Defense-in-Depth

**What it is (Simple Definition):**
Defense-in-depth protects data using multiple layered security mechanisms, so if one layer is breached, another is already in place to limit further exposure.

**The Everyday Analogy:**
Think of defense-in-depth like the layers of a castle: a moat, then a wall, then guards, then a locked vault at the center. Breaching one layer doesn't automatically expose everything inside — each layer slows the attacker down further.

**The seven layers, from outside in:**

| Layer | Focus |
|---|---|
| **Physical security** | Protecting computing hardware in the datacenter — the first line of defense |
| **Identity and access** | Controlling access to infrastructure and change control, using SSO, MFA, and auditing |
| **Perimeter** | DDoS protection and firewalls to filter large-scale attacks before they cause outages |
| **Network** | Limiting communication between resources through segmentation and access controls |
| **Compute** | Securing access to virtual machines and keeping systems patched |
| **Application** | Ensuring apps are secure and free of vulnerabilities, storing secrets securely (e.g., Azure Key Vault) |
| **Data** | Controlling access to operational and customer data, protected with encryption at rest and in transit |

**Why it matters:**
- No single layer is relied upon alone — each slows an attack and provides information security teams can act on.
- Azure provides tools and features at every layer of this model.

**Exam Takeaway:** Defense-in-depth treats data as the center of concentric protective layers — attackers are ultimately after data, whether it's in a database, on a VM disk, in a SaaS app, or in cloud storage.

## 8. Encryption and Key Management in Azure

**What it is (Simple Definition):**
Encryption protects data confidentiality by making it unreadable to unauthorized users, whether the data is stored or moving between systems.

**The Everyday Analogy:**
Think of encryption like sending a message in a locked box that only the intended recipient has the key to — even if someone intercepts the box in transit, they can't read what's inside.

**Two key forms:**
- **Encryption at rest:** Protects data while stored (databases, disks, storage accounts).
- **Encryption in transit:** Protects data while it moves between services, applications, and users.

> **Practical example:** A line-of-business app storing customer records in Azure Storage and Azure SQL should encrypt data both while stored and while moving between application tiers, APIs, and user devices.

### Azure Key Vault

**What it is:** A service for securely storing and controlling access to secrets (connection strings, passwords), encryption keys, and certificates.

**Why it matters (Key Benefit):**
- Centralizes secret and key management instead of storing sensitive values in application code or config files.
- Lets you control who can access keys and secrets, rotate and update them over time, and audit their usage.
- Supports separating duties — limiting who can view, use, or rotate keys.
- Key rotation policies and alerting help avoid service disruption from expiring keys.
- Applications can retrieve secrets at runtime using **managed identities**, reducing hard-coded credentials.

**Exam Takeaway:** Azure Key Vault is the primary Azure service for secure key and secret storage — remember this pairing for the exam.

## 9. Microsoft Defender for Cloud

**What it is (Simple Definition):**
Defender for Cloud is a security posture management and threat protection service, monitoring cloud, on-premises, hybrid, and multicloud resources, and providing recommendations and alerts to improve security.

**The Everyday Analogy:**
Think of it like a building's central security office monitoring cameras and access points across multiple sites at once — flagging suspicious activity and recommending where to add more locks or lighting.

**Why it matters (Key Benefit):**
- Natively integrated into Azure and quick to enable.
- Extends visibility to hybrid and multicloud environments through a single control plane.
- Azure Arc can extend Defender plans to non-Azure machines; Cloud Security Posture Management (CSPM) can assess multicloud resources agentlessly.

**Azure-native protections:**
- **Azure PaaS services:** Detects threats across App Service, Azure SQL, Azure Storage.
- **Azure data services:** Provides security assessments and recommendations for services like SQL and Storage.
- **Networks:** Reduces brute-force exposure via just-in-time VM access and restrictive port policies.

**Hybrid and multicloud coverage:**
- Deploy Azure Arc to extend protection to on-premises machines.
- Defender for Cloud can also protect AWS and GCP resources — extending CSPM recommendations, Defender for Containers protections to Amazon EKS, and Defender for Servers protections to EC2 instances.

### The Three Pillars: Assess, Secure, Defend

| Pillar | What It Covers |
|---|---|
| **Continuously assess** | Vulnerability assessment for VMs, container registries, and SQL servers; integrates with Microsoft Defender for Endpoint for vulnerability findings |
| **Secure** | Builds on Azure Policy to apply controls at management group, subscription, or tenant level; evaluates new resources against the Microsoft cloud security benchmark (MCSB) and calculates a secure score |
| **Defend** | Detects and resolves threats via security alerts and advanced threat protection |

**Security alerts:**
- Describe details of the affected resource.
- Suggest remediation steps.
- Can trigger a logic app in response.
- Use kill-chain analysis to correlate related alerts, showing the full story of an attack.

**Advanced threat protection:** Covers VMs, SQL databases, containers, web applications, and networks, including features like just-in-time VM access and adaptive application controls.

**Exam Takeaway:** Defender for Cloud's three core functions are to assess (know your posture), secure (harden resources against the MCSB), and defend (detect and respond to threats).

## Quick Module Recap

- **Microsoft Entra ID** is Azure's identity and access management service; **Entra Connect** syncs with on-premises AD, while **Entra Domain Services** provides managed domain controllers.
- **Authentication methods** range from passwords to MFA to passwordless — passwordless offers the best mix of security and convenience.
- **External identities** let outside partners and consumers access approved resources using their own credentials.
- **Conditional Access** makes access decisions based on real-time signals like location and device.
- **Azure RBAC** assigns roles at a scope (resource, resource group, subscription, or management group), inheriting downward and combining permissions.
- **Zero Trust** verifies every request regardless of network location; **defense-in-depth** layers multiple protections around your data.
- **Encryption** (at rest and in transit) plus **Azure Key Vault** protect data confidentiality and centralize secret management.
- **Microsoft Defender for Cloud** continuously assesses, secures, and defends resources across Azure, hybrid, and multicloud environments.
