# Module 7: Azure Storage Services

This module covers how Azure stores your data, keeps it safe through redundancy, and how you actually get data into and out of Azure. The redundancy options (LRS, ZRS, GRS, GZRS) and the "9s of durability" numbers are exam favorites, so pay close attention to those tables.

---

## 1. Azure Storage Accounts

**What it is (Simple Definition):**
A storage account gives you a unique namespace in Azure for your data, accessible from anywhere over HTTP or HTTPS. Data inside it is secure, highly available, durable, and massively scalable.

**The Everyday Analogy:**
Think of a storage account like a personal PO Box with a unique address. No matter where you are in the world, mail (your data) can find its way to that exact box, and only you decide who else gets a key.

**Why it matters (Key Benefit):**
- Gives you a globally unique, accessible namespace for your data.
- The account type you choose determines which services and redundancy options are available.

**Storage account types:**

| Type | Supported Services | Redundancy Options | Best For |
|---|---|---|---|
| **Standard general-purpose v2** | Blob Storage (incl. Data Lake Storage), Queue, Table, Azure Files | LRS, GRS, RA-GRS, ZRS, GZRS, RA-GZRS | Most everyday scenarios using Azure Storage |
| **Premium block blobs** | Blob Storage (incl. Data Lake Storage) | LRS, ZRS | High transaction rates, smaller objects, low latency needs |
| **Premium file shares** | Azure Files | LRS, ZRS | High-scale/high-performance apps needing SMB and NFS support |
| **Premium page blobs** | Page blobs only | LRS | Page blob-specific workloads |

**Naming rules:**
- Must be 3 to 24 characters, numbers and lowercase letters only.
- Must be unique across all of Azure.

**Endpoint formats:**

| Storage Service | Endpoint |
|---|---|
| Blob Storage | `https://<storage-account-name>.blob.core.windows.net` |
| Data Lake Storage Gen2 | `https://<storage-account-name>.dfs.core.windows.net` |
| Azure Files | `https://<storage-account-name>.file.core.windows.net` |
| Queue Storage | `https://<storage-account-name>.queue.core.windows.net` |
| Table Storage | `https://<storage-account-name>.table.core.windows.net` |

**Exam Takeaway:** Standard general-purpose v2 is the recommended default account type for most scenarios — reach for a Premium type only when you have specific performance needs.

## 2. Azure Storage Redundancy

**What it is (Simple Definition):**
Azure Storage automatically keeps multiple copies of your data to protect against hardware failures, outages, and regional disasters. The redundancy option you choose balances cost against availability and durability.

**The Everyday Analogy:**
Think of redundancy options like keeping copies of an important document. Locally redundant storage is like keeping three copies in the same filing cabinet. Geo-redundant storage is like keeping copies in the cabinet *and* mailing a copy to a friend in another city, in case your building burns down.

**Key decision factors:**
- How data is replicated in the primary region.
- Whether data is also replicated to a distant secondary region.
- Whether your application needs read access to the secondary region's data.

### Redundancy in the Primary Region

Data is always replicated three times in the primary region, using one of these two options:

| Option | How It Works | Durability | Notes |
|---|---|---|---|
| **Locally redundant storage (LRS)** | Replicates data three times within a single datacenter | At least 11 nines (99.999999999%) | Lowest cost; protects against rack/drive failure but not a datacenter-level disaster |
| **Zone-redundant storage (ZRS)** | Replicates data synchronously across three availability zones in the primary region | At least 12 nines (99.9999999999%) | Data stays available for read/write even if one zone goes down; recommended for high availability within a region |

### Redundancy in a Secondary Region

For extra durability, data can also replicate to a secondary region, geographically distant from the primary (Azure assigns this automatically based on region pairs).

| Option | How It Works | Durability |
|---|---|---|
| **Geo-redundant storage (GRS)** | LRS in the primary region, replicated asynchronously to LRS in the secondary region | At least 16 nines |
| **Geo-zone-redundant storage (GZRS)** | ZRS in the primary region, replicated asynchronously to LRS in the secondary region | At least 16 nines (99.99999999999999%) |

> **Important:** Secondary-region data isn't readable by default unless a failover occurs. Because replication to the secondary region is asynchronous, there's a small risk of data loss — measured by the **recovery point objective (RPO)**, which is typically under 15 minutes for Azure Storage (though there's no SLA guaranteeing replication time).

**Reading secondary-region data before failover:**
Enable **read-access geo-redundant storage (RA-GRS)** or **read-access geo-zone-redundant storage (RA-GZRS)** if your application needs to read from the secondary region without waiting for a failover. Just remember that secondary-region data may not be fully up to date.

**Exam Takeaway:** LRS and ZRS protect within a single region; GRS and GZRS add protection across regions. Add "RA-" to GRS or GZRS if you need read access to the secondary region before failover.

## 3. Azure Storage Services

**What it is (Simple Definition):**
Azure Storage is made up of several core data services, each designed for a different kind of data.

**The Everyday Analogy:**
Think of Azure Storage like a well-organized storage facility: one section for loose boxes of miscellaneous items (Blobs), one for shared filing cabinets (Files), one for a message drop-box between departments (Queues), one for hard drive shelving (Disks), and one for a spreadsheet-style card catalog (Tables).

**Core services at a glance:**

| Service | What It's For |
|---|---|
| **Azure Blobs** | Massively scalable object storage for text/binary data, including big data analytics via Data Lake Storage Gen2 |
| **Azure Files** | Managed file shares for cloud or on-premises use |
| **Azure Queues** | Messaging store for reliable communication between application components |
| **Azure Disks** | Block-level storage volumes for Azure VMs |
| **Azure Tables** | NoSQL store for structured, non-relational data |

**Why it matters (Key Benefit) — Azure Storage overall:**
- **Durable and highly available:** Redundancy options protect against hardware failure and outages.
- **Secure:** All data is encrypted, with fine-grained access control.
- **Scalable:** Built to handle massive data and performance needs.
- **Managed:** Azure handles hardware maintenance and updates.
- **Accessible:** Reachable globally via REST APIs, SDKs, CLI, PowerShell, the Azure portal, or Storage Explorer.

### Azure Blobs

Unstructured object storage for large volumes of text or binary data — ideal for images/documents served to browsers, distributed file access, streaming video/audio, backups, and analytics datasets. Accessible via URLs, REST API, PowerShell, CLI, or client libraries (available for .NET, Java, Node.js, Python, PHP, Ruby).

**Blob access tiers:**

| Tier | Best For |
|---|---|
| **Hot** | Frequently accessed data (e.g., website images) |
| **Cool** | Infrequently accessed data, stored at least 30 days (e.g., customer invoices) |
| **Cold** | Infrequently accessed data, stored at least 90 days |
| **Archive** | Rarely accessed data, stored at least 180 days, flexible latency (e.g., long-term backups) |

**Considerations:**
- Hot, cool, and cold can be set at the account level; archive cannot.
- All four tiers can be set at the blob level.
- Cool and cold have lower storage costs but higher access costs and lower availability SLAs.
- Archive has the lowest storage cost, but the highest rehydration latency and access cost.

### Azure Files

Fully managed cloud file shares accessible via SMB or NFS protocols, mountable concurrently from cloud or on-premises deployments. SMB shares work on Windows, Linux, and macOS; NFS shares work on Linux and macOS. SMB shares can also be cached locally on Windows Servers using **Azure File Sync** for faster access.

**Key benefits:** shared access via standard protocols, fully managed (no server hardware or patching), scriptable via CLI/PowerShell/portal/Storage Explorer, built for high availability, and compatible with standard file I/O APIs plus Azure SDKs and REST APIs.

### Azure Queues

Stores large numbers of messages (up to 64 KB each, millions of messages total) for asynchronous processing, accessed via authenticated HTTP/HTTPS calls. Often paired with **Azure Functions**, where an incoming message triggers a background action.

### Azure Disks

Managed, block-level storage volumes for Azure VMs — virtualized and managed by Azure for improved resiliency and simpler operations.

### Azure Tables

A NoSQL store for large amounts of structured, non-relational data, accessible through authenticated calls from cloud and hybrid environments.

**Exam Takeaway:** Match the data type to the service — unstructured objects go to Blobs, shared file access goes to Files, async messaging goes to Queues, VM storage goes to Disks, and structured non-relational data goes to Tables.

## 4. Azure Data Migration Options

**What it is (Simple Definition):**
Azure provides tools to move existing infrastructure, applications, and data into the cloud — either over the network in real time, or physically shipped for very large datasets.

**The Everyday Analogy:**
Moving into a new house: for regular boxes, you drive back and forth yourself (network-based migration). But if you have a whole houseful of furniture, it's faster to hire a moving truck to haul everything at once (physical data transfer).

### Azure Migrate

A hub that helps manage the assessment and migration of your on-premises environment to Azure.

**What it offers:**
- **Unified migration platform:** A single portal to start, run, and track your migration.
- **Range of tools:** Includes Azure Migrate: Discovery and assessment, and Azure Migrate: Server Migration, plus integration with other Azure services and ISV tools.
- **Integrated tools:** Covers discovery/assessment, server migration, database migration, and web app migration.

### Azure Data Box

A physical migration service for transferring large amounts of data quickly and securely, using a rugged, shippable storage device with up to 80 terabytes of usable capacity.

**How it works:**
1. Order the Data Box through the Azure portal.
2. Set it up locally via the web UI and connect it to your network.
3. Transfer data in or out.
4. Ship the device back — if importing, data uploads automatically once Microsoft receives it.
5. The entire process is tracked end-to-end in the Azure portal.

**Best for:** one-time bulk migrations, periodic large uploads where online transfer is too slow, and exporting large datasets for recovery or regulatory needs.

> **Security note:** Once import data is uploaded (or export data is delivered), the device's disks are wiped according to NIST 800-88r1 standards.

**Exam Takeaway:** Use Azure Migrate for assessing and migrating infrastructure/apps; use Azure Data Box when network bandwidth can't handle a very large one-time data transfer.

## 5. Azure File Movement Options

**What it is (Simple Definition):**
Beyond large-scale migration tools, Azure also offers lightweight tools for moving or syncing individual files or smaller file groups.

**The Everyday Analogy:**
If Azure Migrate and Data Box are moving trucks, these tools are more like a hand-cart — perfect for smaller, everyday jobs like grabbing a few files or keeping folders in sync.

| Tool | What It Does |
|---|---|
| **AzCopy** | A command-line utility for uploading, downloading, copying between storage accounts, and one-directional synchronizing of blobs/files (even across cloud providers) |
| **Azure Storage Explorer** | A graphical app (Windows, macOS, Linux) for managing files and blobs, using AzCopy on the backend |
| **Azure File Sync** | Centralizes file shares in Azure Files while keeping a local Windows file server bi-directionally synced |

**Azure File Sync key features:**
- Access data locally via SMB, NFS, or FTPS.
- Deploy as many local caches worldwide as needed.
- Replace a failed local server by installing Azure File Sync on a new one in the same datacenter.
- Configure cloud tiering, so frequently accessed files stay local while infrequently accessed files remain in the cloud until requested.

**Exam Takeaway:** AzCopy syncs in one direction only; Azure File Sync is the option that keeps data synced bi-directionally between on-premises and the cloud.

## Quick Module Recap

- **Storage accounts** give you a globally unique namespace; account type determines available services and redundancy options.
- **Redundancy** scales from LRS (single datacenter) to ZRS (zone-level) to GRS/GZRS (cross-region), with "RA-" variants adding readable secondary-region access.
- **Core services:** Blobs (unstructured objects), Files (managed shares), Queues (messaging), Disks (VM storage), Tables (NoSQL structured data).
- **Blob access tiers** (Hot, Cool, Cold, Archive) trade off storage cost against access cost and latency.
- **Azure Migrate** handles infrastructure/app migration; **Azure Data Box** handles large-scale physical data transfer.
- **AzCopy, Storage Explorer, and Azure File Sync** handle smaller-scale file movement and syncing.
