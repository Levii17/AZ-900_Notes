# AZ-900 Exam Skills Checklist

This is the official Microsoft skills-measured outline for AZ-900, reorganized into a scannable checklist. Use it as your final review pass — each bullet has a one-line memory-jogger, and a pointer to the full module where you can go deeper if a topic still feels shaky.

---

## Domain 1: Describe Cloud Concepts (25–30%)

### Describe Cloud Computing

- [ ] **Define cloud computing** — Renting IT services (compute, storage, networking, AI) over the internet instead of owning hardware. *(Module 1)*
- [ ] **Describe the shared responsibility model** — You always own data, devices, and identities; the provider always owns the physical datacenter; everything else shifts by service model. *(Module 1)*
- [ ] **Define cloud models: public, private, hybrid** — Public = shared/rented, Private = owned/dedicated, Hybrid = both connected. *(Module 1)*
- [ ] **Identify use cases for each cloud model** — Private for strict control/compliance; public for flexibility and no upfront cost; hybrid for mixing sensitive and flexible workloads. *(Module 1)*
- [ ] **Describe the consumption-based model** — Pay only for what you use; classified as OpEx, not CapEx. *(Module 1)*
- [ ] **Compare cloud pricing models** — Pay-as-you-go vs. Reservations vs. savings plans vs. Spot pricing. *(Module 9)*
- [ ] **Describe serverless** — Event-driven compute (Azure Functions) where infrastructure is fully abstracted away and you pay only for execution time. *(Module 5)*

### Describe the Benefits of Using Cloud Services

- [ ] **High availability and scalability** — Staying up through disruption (SLAs) + adjusting resources to match demand (vertical = bigger/smaller, horizontal = more/fewer). *(Module 2)*
- [ ] **Reliability and predictability** — Recovering from failure via global design + forecasting performance and cost. *(Module 2)*
- [ ] **Security and governance** — Built-in tools to enforce standards and defend against threats like DDoS. *(Module 2)*
- [ ] **Manageability** — "Of the cloud" (automated features) vs. "in the cloud" (portal, CLI, APIs, PowerShell). *(Module 2)*

### Describe Cloud Service Types

- [ ] **Infrastructure as a Service (IaaS)** — Provider covers hardware/network/physical security; you manage everything else. *(Module 3)*
- [ ] **Platform as a Service (PaaS)** — Provider also manages OS, middleware, dev tools; you focus on app code and data. *(Module 3)*
- [ ] **Software as a Service (SaaS)** — Provider manages nearly everything; you manage data, identity, and device access. *(Module 3)*
- [ ] **Use cases for each** — IaaS: lift-and-shift, custom test/dev. PaaS: app development, analytics/BI. SaaS: email, productivity, finance apps. *(Module 3)*

## Domain 2: Describe Azure Architecture and Services (35–40%)

This is the largest domain — budget your study time accordingly.

### Core Architectural Components

- [ ] **Regions, region pairs, sovereign regions** — Regions group datacenters geographically; region pairs (300+ miles apart) protect against region-wide disaster; sovereign regions (US Gov, China) exist for compliance. *(Module 4)*
- [ ] **Availability zones** — Physically separate datacenters within a region (minimum 3 where supported), each with independent power/cooling/networking. *(Module 4)*
- [ ] **Azure datacenters** — The physical facilities themselves; you never interact with them directly. *(Module 4)*
- [ ] **Resources and resource groups** — A resource is anything you deploy; every resource belongs to exactly one resource group. *(Module 4)*
- [ ] **Subscriptions** — Billing boundary + access control boundary, linked to an Azure account. *(Module 4)*
- [ ] **Management groups** — Sit above subscriptions, nestable up to 6 levels, all folding into one Tenant Root Group. *(Module 4)*
- [ ] **Hierarchy** — Resources → Resource Groups → Subscriptions → Management Groups → Tenant Root Group; policies inherit downward. *(Module 4)*

### Compute and Networking Services

- [ ] **Compare compute types: containers, VMs, functions** — VMs = full OS control (IaaS). Containers = virtualize the app, not the OS (lightweight). Functions = event-driven, serverless, pay-per-execution. *(Module 5)*
- [ ] **VM options: VMs, Scale Sets, Availability Sets, Virtual Desktop** — Scale Sets manage identical load-balanced VM groups; Availability Sets group by update/fault domain; Virtual Desktop delivers cloud-hosted Windows desktops. *(Module 5)*
- [ ] **Resources required for VMs** — Size (family/vCPUs/RAM), storage disks, networking (VNet, public IP, ports). *(Module 5)*
- [ ] **Application hosting options: web apps, containers, VMs** — Azure App Service is the PaaS option for web-oriented hosting without managing infrastructure. *(Module 5)*
- [ ] **Virtual networking: VNets, subnets, peering, DNS, VPN Gateway, ExpressRoute** — VNets isolate/route/filter traffic; peering connects VNets privately (even cross-region); ExpressRoute bypasses the public internet entirely. *(Module 6)*
- [ ] **Public vs. private endpoints** — Public = internet-accessible with a public IP; private = internal-only with a private IP from the VNet's address space. *(Module 6)*

### Storage Services

- [ ] **Compare Azure Storage services** — Blobs (unstructured objects), Files (managed shares), Queues (messaging), Disks (VM storage), Tables (NoSQL). *(Module 7)*
- [ ] **Storage tiers** — Hot (frequent access), Cool (30+ days), Cold (90+ days), Archive (180+ days, cheapest storage, highest retrieval cost/latency). *(Module 7)*
- [ ] **Redundancy options** — LRS (single datacenter, 11 nines) → ZRS (zone-level, 12 nines) → GRS/GZRS (cross-region, 16 nines); add "RA-" for readable secondary-region access. *(Module 7)*
- [ ] **Storage account options and types** — Standard general-purpose v2 is the default; Premium types exist for specific performance needs. *(Module 7)*
- [ ] **File movement: AzCopy, Storage Explorer, Azure File Sync** — AzCopy is one-directional; Azure File Sync is bi-directional between on-premises and the cloud. *(Module 7)*
- [ ] **Migration: Azure Migrate, Azure Data Box** — Migrate handles infrastructure/app assessment and migration; Data Box handles large-scale physical data transfer (up to 80 TB). *(Module 7)*

### Identity, Access, and Security

- [ ] **Directory services: Entra ID, Entra Domain Services** — Entra ID is cloud identity/access management; Domain Services provides managed domain controllers without you running them. *(Module 8)*
- [ ] **Authentication methods: SSO, MFA, passwordless** — Passwordless (Windows Hello, Authenticator app, FIDO2 keys) offers the best mix of security and convenience. *(Module 8)*
- [ ] **External identities** — B2B collaboration, B2B direct connect, and Entra External ID for customers let outside users access resources with their own credentials. *(Module 8)*
- [ ] **Microsoft Entra Conditional Access** — Signal → Decision → Enforcement flow based on identity, location, and device context. *(Module 8)*
- [ ] **Azure RBAC** — Role assignments at a scope (resource, resource group, subscription, management group); hierarchical and additive (allow model). *(Module 8)*
- [ ] **Zero Trust** — Verify explicitly, use least privilege, assume breach; never trust based on network location alone. *(Module 8)*
- [ ] **Defense-in-depth** — Seven layers (physical, identity, perimeter, network, compute, application, data) protecting data at the center. *(Module 8)*
- [ ] **Microsoft Defender for Cloud** — Assess, Secure, Defend across Azure, hybrid, and multicloud resources. *(Module 8)*

## Domain 3: Describe Azure Management and Governance (30–35%)

### Cost Management

- [ ] **Factors affecting costs** — Resource type, consumption, maintenance, geography, subscription type, Azure Marketplace. *(Module 9)*
- [ ] **Pricing calculator** — Estimates costs before deployment; never provisions or charges anything. *(Module 9)*
- [ ] **Cost management capabilities** — Cost analysis (visual exploration), cost alerts (budget/credit/department quota), and budgets (can trigger automation). *(Module 9)*
- [ ] **Purpose of tags** — Metadata for organizing, reporting, and governing resources; doesn't auto-inherit between levels. *(Module 9)*

### Governance and Compliance

- [ ] **Microsoft Purview** — Unified data governance and risk/compliance visibility across on-premises, multicloud, and SaaS data. *(Module 10)*
- [ ] **Azure Policy** — Enforces/audits compliance rules, inherited downward; initiatives group related policies toward a broader goal. *(Module 10)*
- [ ] **Resource locks** — Delete lock (can't delete) vs. ReadOnly lock (can't modify or delete); apply on top of RBAC, not instead of it. *(Module 10)*

### Managing and Deploying Resources

- [ ] **Azure portal** — Web-based visual console, resilient across every Azure datacenter. *(Module 11)*
- [ ] **Cloud Shell, Azure CLI, Azure PowerShell** — Cloud Shell runs either shell in the browser with no install; CLI uses Bash syntax, PowerShell uses cmdlets — functionally equivalent. *(Module 11)*
- [ ] **Azure Arc** — Extends Azure governance/management to hybrid and multicloud resources without moving them into Azure. *(Module 11)*
- [ ] **Infrastructure as code (IaC)** — Managing infrastructure through code/templates instead of manual configuration. *(Module 11)*
- [ ] **Azure Resource Manager (ARM) and ARM templates** — The single management layer behind every Azure request; ARM templates (JSON) and Bicep (simpler syntax) both describe desired state declaratively. *(Module 11)*

### Monitoring Tools

- [ ] **Azure Advisor** — Personalized recommendations across 5 categories: reliability, security, performance, operational excellence, cost. *(Module 12)*
- [ ] **Azure Service Health** — Three scopes: Azure Status (global) → Service Health (your environment) → Resource Health (single resource). *(Module 12)*
- [ ] **Azure Monitor: Log Analytics, alerts, Application Insights** — Monitor collects data; Log Analytics queries it; alerts notify on conditions (metric- or log-based); Application Insights focuses on web app performance. *(Module 12)*

## Final Exam-Day Reminders

- Domain 2 (Azure Architecture and Services) carries the most weight — make sure compute, networking, storage, and identity are rock-solid.
- Watch for scenario questions that ask "which service/model fits this situation" rather than pure definitions — the exam leans heavily on applied understanding.
- Comparison tables (IaaS/PaaS/SaaS, LRS/ZRS/GRS/GZRS, VM families, redundancy durability numbers) are common testing points — review those tables one more time before you sit the exam.
